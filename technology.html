FastAPI
v0.115.5 · Python 3.11
Async REST API Framework
FastAPI provides the async-first REST API layer for LogiTwin. All 23 endpoints are fully async — meaning hundreds of concurrent simulation requests never block each other. Auto-generated OpenAPI docs at /docs. Pydantic v2 integration means every request and response is validated at the Python level before touching the database.
✓Native async/await — no blocking I/O anywhere in the request chain
✓Pydantic v2 strict mode catches malformed payloads before Neo4j sees them
✓Auto OpenAPI schema generation — client integrations start from the /docs page

Neo4j
v5.20.0 · Docker
Graph Database — The Core
Supply chains are graphs. A supplier ships to a factory through a port — that's a 3-node, 2-relationship traversal. In a relational database that's a JOIN across 3 tables with foreign keys. In Neo4j it's a single Cypher MATCH that runs in milliseconds regardless of how many nodes exist. When Hamburg port goes offline, finding every affected factory is one BFS query — not a 5-table JOIN.
✓BFS traversal in milliseconds — SQL would need recursive CTEs and multiple JOINs
✓Native relationship properties — lead_time_days, transit_days stored on edges
✓MERGE pattern — re-ingesting never creates duplicate nodes

Relationships55+
Driverneo4j async

Celery + Redis
Celery 5.4 · Redis 7
Async Task Queue
Simulations can't run synchronously — a single simulation traverses 55+ graph relationships and writes back to Neo4j. Running that on the FastAPI request thread would block all other API calls. Celery workers run as completely separate OS processes, each creating their own Neo4j driver. Redis is both the message broker (task queue) and result backend (where results are stored for polling).
✓Workers are separate processes — FastAPI never waits for a simulation to complete
✓4 concurrent workers — 4 simulations can run in parallel
✓task_acks_late=True — task marked complete only after successful Neo4j write
Concurrency4 workers
Brokerredis://6379
Retries3 max
Backoff5s exponential
Pydantic v2
v2.10 · Strict mode
Data Validation Layer
Every piece of data entering LogiTwin goes through Pydantic v2 strict validation before touching Neo4j. A supplier without an ID, a factory with capacity_rate > 1.0, a relationship with negative lead_time_days — all rejected at the API boundary with clear error messages. Pydantic v2 is 5-50x faster than v1 and uses Rust under the hood for core validation.
✓Strict mode — no silent coercion, wrong types always raise ValidationError
✓model_config="str_strip_whitespace" — client whitespace bugs never hit the DB
✓BaseSettings for config — all secrets loaded from .env, never hardcoded
Models14+
ModeStrict
ConfigBaseSettings
Speed5-50x v1
HTTPX
v0.28.1 · Async
Carrier API Hook Client
When LogiTwin fires a mitigation hook, it uses HTTPX — the async HTTP client built for Python's asyncio. When multiple hooks fire in parallel (Maersk + DHL + FedEx simultaneously), asyncio.gather runs them all concurrently. One slow or failed carrier API never blocks others. 10-second timeout, 3 retries with exponential backoff, return_exceptions=True isolation.
✓Fully async — fires 6 carrier hooks in parallel without blocking
✓Full exception isolation — one timeout never cancels other hooks
✓Custom headers — X-LogiTwin-Hook, X-Action-Type sent to carrier APIs
Timeout10s
Retries3 max
Hooks6 registered
Patterngather + isolation
Engineering Principles

Three rules. Never broken.

Async-First
Every network call, every database transaction, every simulation — fully async/await. The FastAPI request thread never blocks waiting for Neo4j or Celery. Uvicorn handles concurrent requests on a single event loop. Workers use asyncio.run() to bridge sync Celery tasks into async Neo4j operations.
Exception Isolation
Every downstream connector is wrapped in structured exception handling. HTTPX carrier hooks use asyncio.gather(return_exceptions=True) — one failed Maersk connection never cancels the DHL hook. Neo4j errors in Celery workers are caught, logged, and returned as structured error dicts — never stack traces to the client.
Decoupled Config
Zero secrets in code. Every URI, key, password, and parameter lives in .env files loaded by Pydantic BaseSettings. NEO4J_URI, REDIS_URL, JWT_SECRET_KEY, JWT_EXPIRE_MINUTES — all configurable without touching source code. Rotating credentials in production is a one-line .env change and a process restart.
